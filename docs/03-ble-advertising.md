# 03 — BLE advertising from scratch

*Everything that happens between `ble_build_adv()` and your iPhone showing
"FindMyKeys −52 dBm".*

Relevant sources: [`src/radio.c`](../src/radio.c), [`src/ble.c`](../src/ble.c),
[`src/ficr.c`](../src/ficr.c), [`src/main.c`](../src/main.c).

## The 40 channels, and why we only use 3

BLE divides the 2.4 GHz ISM band into 40 channels, 2 MHz apart. Channels
0–36 carry connection data. Channels **37, 38, 39** are the *primary
advertising channels* — and they are deliberately spread across the band:

```
2402       2426                                   2480 MHz
 │37│ 0..10 │38│ 11..36                            │39│
  ▲          ▲                                      ▲
  └──────────┴── parked around Wi-Fi channels 1/6/11's gaps ─┘
```

A scanner (your phone) listens on one advertising channel at a time,
hopping. An advertiser transmits the same packet on all three in quick
succession — one *advertising event* — so they meet no matter where the
scanner happens to be listening. That's the `for` loop over
`{37, 38, 39}` in `main.c`, and it's why `radio_send()` takes a channel:
each hop needs a retune (`FREQUENCY`) and a re-seed of the whitener
(`DATAWHITEIV`, below).

The interval between events is 100 ms here (20 ms–10.24 s is legal for our
PDU type), **plus 0–10 ms of random jitter** — required by the spec so two
beacons that power on simultaneously don't collide forever. The jitter
comes from the hardware RNG (`rng.c`); the interval sleep is the RTC
(doc 02 territory: WFE until the COMPARE interrupt).

## What's actually on the air

One advertising packet at 1 Mbit/s, ~44 bytes, ~350 µs:

```
┌──────────┬──────────────────┬─────────────────────────────┬─────────┐
│ preamble │  access address  │            PDU              │  CRC    │
│  1 byte  │     4 bytes      │        2 + 6 + 28 bytes     │ 3 bytes │
│   0xAA   │   0x8E89BED6     │  (that's OUR byte array)    │ by HW   │
└──────────┴──────────────────┴─────────────────────────────┴─────────┘
                                        └── whitened ───────────────┘
```

- **Preamble** `10101010` — lets the receiver's clock recovery lock on.
  Generated by the radio hardware.
- **Access address** — a fixed magic number, `0x8E89BED6`, shared by *all*
  BLE advertising on Earth. Scanners correlate against it to find packet
  starts. (Connections negotiate private access addresses; advertising uses
  this public one.) In `radio.c` it's split across `BASE0`/`PREFIX0` — the
  nRF radio composes logical addresses from a base + prefix scheme.
- **PDU** — the part we build byte-by-byte in `ble.c`. Detailed below.
- **CRC** — 24 bits, polynomial `0x65B`, init `0x555555`, computed over the
  PDU only. We configure it (`CRCCNF/CRCPOLY/CRCINIT`) and the hardware
  computes and appends it. Get any of those three values wrong and every
  phone silently drops your packets — the most common "why does nothing
  appear" failure.

### Whitening

Long runs of identical bits (imagine a payload full of `0x00`) upset radio
receivers — clock recovery drifts and the DC bias shifts. BLE therefore
XORs the PDU+CRC with a pseudorandom bit stream from a 7-bit LFSR seeded
with the **channel index**. The receiver runs the same LFSR to undo it.
That seed is why `DATAWHITEIV` must be rewritten on every hop: a packet
whitened for channel 37 is garbage if decoded with channel 39's sequence.

## The PDU we build (`ble.c`)

```
offset  content
  0     header  = 0x42
             bits [3:0] = 0x2  → ADV_NONCONN_IND (non-connectable broadcast)
             bit  [6]   = 1    → TxAdd: the address below is *random*, not public
  1     length  = 34            (everything after this byte)
  2..7  AdvA    = our 6-byte address, least significant byte first
  8..   AdvData = a sequence of AD structures:      [len][type][payload...]
        ┌ 02 01 06             Flags: LE General Discoverable, no BR/EDR
        ├ 0B 09 "FindMyKeys"   Complete Local Name (what the phone displays)
        └ 0C FF FF FF 01 <ctr:4> <tag:4>
              │  └┬─┘  │    │       └── SipHash tag        (doc 04)
              │   │    │    └── rolling counter, LE
              │   │    └── version byte of our format
              │   └── company id 0xFFFF = "internal use / testing"
              └── AD type 0xFF = Manufacturer Specific Data
```

Three details that bite people:

1. **ADV_NONCONN_IND** means we never listen. There's no scan response, no
   connection — pure broadcast. It's the simplest legal PDU and the honest
   choice for a beacon (it's also why the phone's "connect" button will
   fail — that's correct behavior).
2. **Everything is little-endian on air**, including the address. The
   `e2:2a:8f:...` a scanner displays is the *reverse* of the byte order in
   the packet.
3. **31-byte AdvData budget.** Name + flags + 11 bytes of manufacturer data
   just fits. This is why real products truncate names — every byte here is
   contested real estate.

## Where the address comes from (`ficr.c`)

Nordic burns a random 64-bit ID into each chip's **FICR** (Factory
Information Configuration Registers) at manufacture — read-only memory at
`0x10000000`. `DEVICEADDR[0]`/`DEVICEADDR[1]` give 6 bytes of it: a
per-chip random MAC, no EEPROM or provisioning step needed. This answers
"how does my board know its own address": it was born with it.

One BLE rule: addresses come in types, flagged by the top two bits. A
**random static** address must have both top bits set (`0b11...`). We OR
`0xC0` into the top byte, and correspondingly set TxAdd=1 in the PDU
header. Type and bits must agree, or scanners discard the packet.

## Driving the RADIO peripheral (`radio.c`)

The nRF radio is built around three ideas — once you see them, the
datasheet reads easily:

**Tasks and events.** You start actions by writing 1 to a TASKS register
(`TASKS_TXEN`); hardware reports state by setting EVENTS registers
(`EVENTS_READY`, `EVENTS_END`, `EVENTS_DISABLED`) which you clear yourself.
This is the idiom for *every* nRF peripheral (compare `clock.c`, `rng.c`).

**Shorts.** A "short" hard-wires an event to a task inside the peripheral,
zero CPU involved. We set two:

```
READY ──▶ START      (transmit the instant ramp-up finishes)
END   ──▶ DISABLE    (power down the instant the packet ends)
```

So `radio_send()` is: retune, point DMA at the buffer, clear events, fire
`TASKS_TXEN`, and wait for `EVENTS_DISABLED`. One register write launches
the whole TX ramp → transmit → shutdown sequence.

**EasyDMA.** `PACKETPTR` holds a *RAM* address; the radio fetches bytes
itself. Flash addresses silently transmit garbage — that's why `s_pdu`
in `main.c` is a global (RAM), never a `const` array (flash).

The one-time setup in `radio_init()` is exactly the on-air format from the
previous section, expressed as registers: `MODE` = BLE 1 Mbit; `PCNF0` says
"1-byte S0 + 8-bit length field" (matching our header layout); `PCNF1` says
"max 37 payload bytes, 3+1-byte access address, little-endian, whitening
on"; plus the CRC trio and the access address. Every line in that function
maps to one row of the packet diagram — read them side by side and the
"magic BLE config" evaporates.

## Timing reality check

At 1 Mbit/s our ~44-byte packet is ~350 µs on air; three channels plus
ramp-ups ≈ 1.2 ms of radio time per event. At a 100 ms interval the radio
is off >98% of the time, and the CPU sleeps (WFE) between events. That
duty cycle is the entire story of multi-year coin-cell beacons — the DK
just spends the savings on its debug circuitry instead.
